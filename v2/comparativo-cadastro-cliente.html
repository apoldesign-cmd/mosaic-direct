Por que Sidenav vence — critério por critério

Critério Modal central (atual) Sidenav (SLDS Panel) (recomendado)
Preservação de contexto ⚠ Bloqueia Tabela "Ship-To Accounts" some atrás do overlay. Usuário não pode conferir se já tem cliente parecido cadastrado enquanto preenche. ✓ Mantém Tabela permanece visível à esquerda. Possível identificar duplicatas, copiar CEP, comparar antes de salvar.
Altura & footer ⚠ Quebra Modal cresce com conteúdo. Em laptop de 13" o botão "Add New Customer" fica abaixo da viewport — usuário não vê CTA. Exatamente o que o print mostra. ✓ Estável Altura fixa = viewport. Header + Body scrollable + Footer fixo. Botão "Salvar" sempre visível, qualquer monitor.
Scroll & hierarquia ⚠ Todo bloco O modal inteiro scrolla — título e ações somem. Usuário se perde no formulário. ✓ Só o body Header (título) e footer (ações) permanecem fixos. Hierarquia preservada.
Densidade de campos ⚠ 3 colunas apertadas Pra caber 13 campos sem virar Modal gigante, força grid 3 col. Labels longos quebram, campos curtos ficam estranhos. ✓ 2 colunas + seções Layout vertical natural. Agrupamento em "Dados / Endereço / Fiscal" reduz carga cognitiva e melhora preenchimento.
Acessibilidade ⚠ Focus trap rígido Em formulários longos, leitor de tela navega 13 campos sem orientação. WCAG 2.4.3 (focus order) sofre. ✓ Foco gerenciado Panel permite live regions e seções marcadas. Foco previsível, suporta navegação por landmarks.
Responsivo ⚠ Mal aproveitado Em tablet o modal vira "quase fullscreen" desperdiçando espaço; em mobile fica claustrofóbico. ✓ Colapsa elegante Em tablet ocupa 60% lateral; em mobile vira full-screen lateral fluido — padrão nativo.
Consistência interna ⚠ Quebra padrão As outras 3 telas do Mosaic Direct (Agendar Normal, Agendar Protocolo, Criar Protocolo) já usam sidenav lateral. Modal aqui é exceção. ✓ Aderente Mesmo padrão visual, mesma posição, mesmo modelo mental do usuário. Reaproveita componente já validado com PO.
Aderência SLDS ⚠ Mau uso Diretriz oficial: "Modals are used to focus the user on a single task". Cadastrar 13 campos não é single task. ✓ Componente certo SLDS Panel dockado é o componente oficial pra workflows de formulário lateral. slds-panel_docked-right.
Implementação Mesmo custo. slds-modal. Mesmo custo. slds-panel slds-panel_docked-right ou lightning-quick-action-panel.
Referência SLDS oficial: Panels → lightningdesignsystem.com/components/panels  ·  Modals → lightningdesignsystem.com/components/modals
Salesforce usa Panel nativamente em: Service Console (Knowledge sidebar), Path Assistant, Utility Bar, Omni-Channel, Field-Level Help, CPQ Quote Line Editor.

Veredito

O Sidenav (SLDS Panel dockado à direita) é objetivamente superior para o fluxo "Adicionar Novo Cliente": preserva o contexto da tabela, mantém footer/CTA sempre visíveis, agrupa campos em seções, aderente ao SLDS, consistente com as outras 3 telas do Mosaic Direct e usado nativamente pela própria Salesforce em múltiplos produtos. Modal central é o componente errado para workflows de formulário multi-campo.
